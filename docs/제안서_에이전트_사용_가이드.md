## 1. 제안서 에이전트란?

### 1.1 개요

제안서 에이전트는 RFP(제안요청서) 문서를 입력받아 **PPTX 형식의 입찰 제안서를 자동 생성**하는 AI 기반 시스템입니다.

### 1.2 핵심 가치: 누구나 기준 이상의 제안서 작성 가능

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   "제안서 에이전트를 구축하면                                         │
│    경험이 없는 신입사원도 기준 이상의 제안서를 작성할 수 있습니다"       │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Before (기존 방식)                                                 │
│   ─────────────────                                                 │
│   • 제안서 품질이 담당자 역량에 크게 의존                              │
│   • 신입/주니어는 선배 도움 없이 작성 불가                             │
│   • 담당자별로 제안서 구조/품질 편차 큼                                │
│   • 우수 제안서 노하우가 개인에게 귀속                                 │
│                                                                     │
│   After (에이전트 도입 후)                                           │
│   ─────────────────────                                             │
│   • 검증된 구조(Impact-8)가 자동 적용                                 │
│   • 레퍼런스 제안서 스타일이 그대로 반영                               │
│   • 누구나 일정 수준 이상의 제안서 생성 가능                           │
│   • 조직의 제안 노하우가 시스템에 축적                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.3 주요 특징

| 특징 | 설명 |
|------|------|
| **표준화된 구조** | Impact-8 Framework로 검증된 제안서 구조 자동 적용 |
| **레퍼런스 스타일** | 수주 성공 제안서의 디자인/구성을 그대로 반영 가능 |
| **품질 평준화** | 담당자 경험과 무관하게 일정 품질 보장 |
| **노하우 축적** | 우수 제안서 패턴이 프롬프트로 문서화되어 조직 자산화 |

## 2. 에이전트 시대의 새로운 업무 방식

> **핵심 변화**: 실무자(Player)에서 디렉터(Director)로
> 디자인과 레이아웃은 AI가 담당하고, 사람은 기획과 디렉팅에 집중합니다.

### 2.1 업무 방식의 패러다임 전환

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    일하는 방식이 완전히 바뀝니다                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   기존 방식: "직접 만드는 사람" (Player)                                 │
│   ════════════════════════════════════                                  │
│                                                                         │
│   담당자가 모든 것을 직접 수행                                           │
│   • 직접 글을 씀                                                        │
│   • 직접 디자인함                                                       │
│   • 직접 레이아웃 구성                                                   │
│   • 직접 차트 제작                                                       │
│                                                                         │
│   ─────────────────────────────────────────────────────────────────     │
│                                                                         │
│   새로운 방식: "방향을 지시하는 사람" (Director)                          │
│   ════════════════════════════════════════════                          │
│                                                                         │
│   담당자는 기획과 디렉팅에 집중                                          │
│   • 전략과 방향 설정                                                     │
│   • AI에게 지시 및 피드백                                                │
│   • 결과물 검토 및 수정 요청                                             │
│   • 최종 품질 관리                                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

```
│                               │                                         │
```

### 2.4 역할의 변화

| 구분 | 기존 역할 | 새로운 역할 |
|------|----------|------------|
| **핵심 업무** | 직접 만들기 | 방향 설정 & 품질 관리 |
| **스킬 중심** | PPT 스킬, 글쓰기 | 전략적 사고, 커뮤니케이션 |
| **시간 사용** | 실무 80% + 기획 20% | 기획 50% + 검토 25% + 디렉팅 20% |
| **경쟁력** | 빠른 손, 디자인 감각 | 수주 전략, 클라이언트 이해 |
| **성장 방향** | 숙련된 실무자 | 전략적 디렉터 |

```
│   □ 경쟁사 대비 차별화 포인트를 설명할 수 있는가?                        │
│                                                                         │
│   □ 직접 만드는 시간보다 생각하는 시간이 더 긴가?                        │
│                                                                         │
│   □ "내가 더 잘 만들 수 있어"라는 생각을 버렸는가?                       │
│                                                                         │
│   □ AI를 도구가 아닌 팀원처럼 활용하고 있는가?                           │
│                                                                         │

```

## 3. LLM 채팅 방식 vs 바이브 코딩 방식

### 3.1 두 가지 AI 활용 방식

AI를 활용한 제안서 작성에는 크게 두 가지 방식이 있습니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                     AI 활용 방식 비교                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1. LLM 채팅 방식 (ChatGPT, Gemini, Claude Web 등)                 │
│   ──────────────────────────────────────────────────                │
│   사람이 AI와 대화하며 텍스트를 받아 수동으로 작업                     │
│                                                                     │
│   2. 바이브 코딩 방식 (Claude Code + [회사명])                    │
│   ──────────────────────────────────────────────────                │
│   AI가 직접 파일을 읽고, 코드를 실행하고, 결과물을 생성                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 LLM 채팅 방식 (ChatGPT, Gemini 등)

#### 작동 방식

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 채팅 방식 워크플로우                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   사용자                          ChatGPT/Gemini                    │
│     │                                  │                            │
│     │  ① "제안서 작성해줘" + RFP 복붙   │                            │
│     │ ─────────────────────────────────>│                            │
│     │                                  │                            │
│     │  ② 텍스트 응답 (마크다운)         │                            │
│     │ <─────────────────────────────────│                            │
│     │                                  │                            │
│     │  ③ 사용자가 수동으로 복사          │                            │
│     ▼                                  │                            │
│   Word/PPT에 붙여넣기                   │                            │
│     │                                  │                            │
│     │  ④ "이 부분 수정해줘"             │                            │
│     │ ─────────────────────────────────>│                            │
│     │                                  │                            │
│     │  ⑤ 수정된 텍스트                  │                            │
│     │ <─────────────────────────────────│                            │
│     │                                  │                            │
│     ▼                                  │                            │
│   다시 수동으로 붙여넣기 (반복...)       │                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 특징

| 항목 | 설명 |
|------|------|
| **입력** | 텍스트 복사/붙여넣기 |
| **출력** | 텍스트 (마크다운) |
| **파일 접근** | 불가 (사용자가 내용 전달) |
| **코드 실행** | 불가 |
| **컨텍스트** | 대화창 내로 제한 |
| **자동화** | 없음 (모든 작업 수동) |

#### 장점
- 별도 설치 없이 웹에서 바로 사용
- 직관적인 대화형 인터페이스
- 다양한 질문에 범용적 응답

#### 단점
- **복사/붙여넣기 반복**: 모든 내용을 수동으로 옮겨야 함
- **파일 생성 불가**: PPTX, 차트 등 직접 생성 불가
- **컨텍스트 손실**: 긴 대화 시 앞부분 내용 잊어버림
- **일관성 부족**: 매번 새로운 대화 시작 시 처음부터 설명 필요
- **프로젝트 이해 불가**: 전체 코드베이스/파일 구조 파악 불가

### 3.3 바이브 코딩 방식 (Vibe Coding)

#### 바이브 코딩이란?

**바이브 코딩(Vibe Coding)**은 자연어로 의도만 전달하면 AI가 직접 코드를 작성하고, 파일을 수정하고, 명령어를 실행하는 방식입니다.

> "코드를 직접 작성하지 않고, 원하는 바(vibe)만 전달하면 AI가 구현한다"

#### 작동 방식

```
┌─────────────────────────────────────────────────────────────────────┐
│                    바이브 코딩 방식 워크플로우                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   사용자                     Claude Code (에이전트)                  │
│     │                              │                                │
│     │  ① "RFP 분석해서 제안서 만들어"│                                │
│     │ ──────────────────────────────>│                                │
│     │                              │                                │
│     │                              ▼                                │
│     │                    ┌─────────────────┐                        │
│     │                    │ 자동 실행       │                        │
│     │                    │ • RFP 파일 읽기 │                        │
│     │                    │ • 내용 분석     │                        │
│     │                    │ • 콘텐츠 생성   │                        │
│     │                    │ • PPTX 생성    │                        │
│     │                    │ • 파일 저장     │                        │
│     │                    └─────────────────┘                        │
│     │                              │                                │
│     │  ② 완료! output/제안서.pptx 생성됨                             │
│     │ <──────────────────────────────│                                │
│     │                              │                                │
│     │  ③ "Phase 4 슬라이드 더 추가해"│                                │
│     │ ──────────────────────────────>│                                │
│     │                              │                                │
│     │                    [자동으로 파일 수정]                        │
│     │                              │                                │
│     │  ④ 수정 완료!                 │                                │
│     │ <──────────────────────────────│                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### 특징

| 항목 | 설명 |
|------|------|
| **입력** | 자연어 명령 |
| **출력** | 실제 파일 (PPTX, PDF, 코드 등) |
| **파일 접근** | 직접 읽기/쓰기 가능 |
| **코드 실행** | 가능 (Python, Shell 등) |
| **컨텍스트** | 프로젝트 전체 파일 이해 |
| **자동화** | 완전 자동화 가능 |

#### 장점
- **End-to-End 자동화**: RFP 입력 → PPTX 출력까지 자동
- **파일 직접 생성**: PPTX, 차트, 다이어그램 자동 생성
- **프로젝트 이해**: 전체 코드베이스와 설정 파일 파악
- **지속적 컨텍스트**: 프로젝트 설정이 항상 유지됨
- **반복 작업 최소화**: 한 번 설정하면 계속 재사용

#### 단점
- 초기 환경 설정 필요
- CLI 기반으로 학습 필요
- API 비용 발생

### 3.6 왜 바이브 코딩 방식인가?

```
┌─────────────────────────────────────────────────────────────────────┐
│                  제안서 에이전트에 바이브 코딩이 필요한 이유           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1. 복잡한 산출물 생성                                              │
│   ─────────────────────                                             │
│   • 70-140페이지 PPTX 파일                                          │
│   • 차트, 조직도, 타임라인 자동 생성                                  │
│   • 레퍼런스 스타일 100% 재현                                        │
│   → ChatGPT로는 텍스트만 가능, 직접 PPTX 생성 불가                   │
│                                                                     │
│   2. 프로젝트 설정 유지                                              │
│   ─────────────────────                                             │
│   • 회사 정보 (company_profile.json)                                │
│   • 디자인 스타일 (design_style.py)                                    │
│   • Phase별 프롬프트 (config/prompts/)                              │
│   → ChatGPT는 매번 새로 설명해야 함                                  │
│                                                                     │
│   3. 반복 작업 자동화                                                │
│   ─────────────────────                                             │
│   • 동일한 구조로 여러 제안서 생성                                    │
│   • RFP만 바꾸면 새 제안서 자동 생성                                  │
│   → ChatGPT는 매번 처음부터 작업                                     │
│                                                                     │
│   4. 품질 일관성 보장                                                │
│   ─────────────────────                                             │
│   • Impact-8 Framework 자동 적용                                    │
│   • 슬라이드 유형별 템플릿 적용                                       │
│   • 누가 사용해도 동일한 품질                                         │
│   → ChatGPT는 사용자마다 결과가 다름                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.7 언제 어떤 방식을 사용할까?

| 상황 | 권장 방식 | 이유 |
|------|----------|------|
| 제안서 아이디어 브레인스토밍 | LLM 채팅 | 빠른 아이디어 교환 |
| RFP 내용 요약/이해 | LLM 채팅 | 간단한 질의응답 |
| 간단한 문구 작성 | LLM 채팅 | 짧은 텍스트 생성 |
| **70p+ 제안서 생성** | **바이브 코딩** | PPTX 직접 생성 필요 |
| **레퍼런스 스타일 적용** | **바이브 코딩** | 스타일 파일 적용 필요 |
| **반복적 제안서 작성** | **바이브 코딩** | 설정 재사용 필요 |
| **팀 전체가 동일 품질** | **바이브 코딩** | 표준화된 시스템 필요 |

### 2.2 Claude Code 설치 및 설정

#### Step 1: Anthropic API 키 발급

1. [Anthropic Console](https://console.anthropic.com/) 접속
2. 회원가입 또는 로그인
3. API Keys 메뉴에서 새 키 생성
4. 생성된 키를 안전한 곳에 저장

#### Step 2: Claude Code CLI 설치

```bash
# npm을 통한 설치 (Node.js 18+ 필요)
npm install -g @anthropic-ai/claude-code

# 설치 확인
claude --version
```

#### Step 3: API 키 설정

```bash
# 환경변수로 설정 (권장)
export ANTHROPIC_API_KEY="your-api-key-here"

# 또는 프로젝트 .env 파일에 설정
echo "ANTHROPIC_API_KEY=your-api-key-here" >> .env
```

### 2.3 프로젝트 설치 및 설정

#### Step 1: 프로젝트 클론 및 의존성 설치

```bash
# 프로젝트 디렉토리로 이동
cd /path/to/입찰제안서프로젝트

# Python 가상환경 생성 (권장)
python -m venv venv

# 가상환경 활성화
# macOS/Linux:
source venv/bin/activate
# Windows:
.\venv\Scripts\activate

# 의존성 설치
pip install -r requirements.txt
```

#### Step 2: 환경 설정 파일 구성

```bash
# .env.example을 복사하여 .env 생성
cp .env.example .env
```

`.env` 파일 내용:

```env
# Anthropic API 설정
ANTHROPIC_API_KEY=sk-ant-your-api-key-here

# Claude 모델 설정 (선택)
CLAUDE_MODEL=claude-sonnet-4-20250514

# 출력 디렉토리 (선택)
OUTPUT_DIR=./output
```

#### Step 3: 설치 확인


## 5. 제안서 에이전트 작동 원리

### 5.1 전체 아키텍처

제안서 에이전트는 **2-Layer 구조**로 설계되어 있습니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    RFP 문서 입력 (PDF/DOCX)                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: Claude Code (콘텐츠 생성)                              │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────────┐  │
│  │ 문서 파싱    │ →  │ RFP 분석    │ →  │ Impact-8 콘텐츠 생성 │  │
│  │ (PDF/DOCX)  │    │ (요구사항)   │    │ (Phase 0~7)         │  │
│  └─────────────┘    └─────────────┘    └─────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
                     ProposalContent (JSON)
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Layer 2: [회사명] (문서화)                                   │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────────┐  │
│  │ 스타일 적용  │ →  │ 슬라이드    │ →  │ 차트/다이어그램     │  │
│  │ (레퍼런스)   │    │ 빌더 실행   │    │ 생성               │  │
│  └─────────────┘    └─────────────┘    └─────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                    PPTX 제안서 출력 (70~140p)                    │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 단계별 처리 과정

#### Step 1: 문서 파싱 (Document Parsing)

```
입력 파일 → PDF Parser / DOCX Parser → 구조화된 텍스트
```

| 파서 | 사용 라이브러리 | 추출 내용 |
|------|---------------|----------|
| PDF Parser | `pypdf`, `pdfplumber` | 텍스트, 테이블, 메타데이터 |
| DOCX Parser | `python-docx` | 텍스트, 스타일, 섹션 |

**관련 파일**: [src/parsers/pdf_parser.py](../src/parsers/pdf_parser.py), [src/parsers/docx_parser.py](../src/parsers/docx_parser.py)

#### Step 2: RFP 분석 (RFP Analysis)

Claude API를 사용하여 RFP 문서에서 핵심 정보를 추출합니다.

```python
# RFPAnalysis 스키마 (주요 필드)
{
    "project_name": "프로젝트명",
    "client_name": "발주처",
    "project_overview": "프로젝트 개요",
    "requirements": {
        "functional": ["기능 요구사항"],
        "technical": ["기술 요구사항"],
        "management": ["관리 요구사항"]
    },
    "evaluation_criteria": ["평가 기준"],
    "deliverables": ["산출물"],
    "timeline": {"start": "시작일", "end": "종료일"},
    "budget": {"amount": 금액, "currency": "KRW"},
    "winning_strategy": "수주 전략",
    "differentiation_points": ["차별화 포인트"]
}
```

**관련 파일**: [src/agents/rfp_analyzer.py](../src/agents/rfp_analyzer.py), [src/schemas/rfp_schema.py](../src/schemas/rfp_schema.py)

#### Step 3: 콘텐츠 생성 (Content Generation)

Impact-8 Framework에 따라 8개 Phase별로 콘텐츠를 생성합니다.

```
RFP 분석 결과 + 회사 데이터 + Phase 프롬프트 → Claude API → PhaseContent
```

각 Phase마다:
1. 해당 Phase의 프롬프트 템플릿 로드 (`config/prompts/phase{N}_{name}.txt`)
2. RFP 분석 결과와 회사 정보를 프롬프트에 주입
3. Claude API 호출하여 슬라이드 콘텐츠 생성
4. JSON 형식의 `PhaseContent` 객체 반환

**관련 파일**: [src/agents/content_generator.py](../src/agents/content_generator.py), [config/prompts/](../config/prompts/)

#### Step 4: PPTX 생성 (PPTX Generation)

생성된 `ProposalContent`를 PPTX 형식으로 변환합니다.

```
ProposalContent → TemplateManager → SlideBuilder → PPTXGenerator → .pptx
```

- **TemplateManager**: 레퍼런스 스타일 설정 로드
- **SlideBuilder**: Phase별 슬라이드 구조 생성
- **ChartGenerator**: 차트 (bar, line, pie, gantt)
- **DiagramGenerator**: 조직도, 프로세스 플로우, 타임라인

**관련 파일**: [src/generators/pptx_generator.py](../src/generators/pptx_generator.py), [src/orchestrators/pptx_orchestrator.py](../src/orchestrators/pptx_orchestrator.py)

### 5.3 사용 방법

```bash
# 기본 사용법
python main.py generate input/rfp.pdf -n "프로젝트명" -c "발주처"

# 제안서 유형 지정 (마케팅/PR)
python main.py generate input/rfp.pdf -n "소셜미디어 운영" -c "[발주처명]" -t marketing_pr

# RFP 분석만 수행
python main.py analyze input/rfp.pdf

# 지원 유형 확인
python main.py types
```

**지원 제안서 유형**:
- `marketing_pr`: 마케팅/PR/소셜미디어
- `event`: 이벤트/행사
- `public`: 공공/입찰
- `it_system`: IT/시스템
- `consulting`: 컨설팅
- `general`: 일반

## 6. 제안서 구성 및 구조

### 6.1 Impact-8 Framework

수주 성공 제안서 분석을 기반으로 개발된 8-Phase 구조입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│  PHASE 0: HOOK (티저)                              3-10p (5%)  │
│  ─────────────────────────────────────────────────────────────  │
│  • 임팩트 있는 오프닝                                           │
│  • 핵심 메시지와 비전 전달                                       │
│  • 비주얼 중심의 스토리텔링                                      │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 1: SUMMARY                                   3-5p (5%)  │
│  ─────────────────────────────────────────────────────────────  │
│  • Executive Summary (의사결정자용 5분 요약)                     │
│  • 제안 핵심 내용 압축                                          │
│  • 예상 성과 및 투자 대비 효과                                   │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 2: INSIGHT                                  8-15p (10%) │
│  ─────────────────────────────────────────────────────────────  │
│  • 시장 환경 분석                                               │
│  • 문제 정의 및 기회 도출                                        │
│  • 숨겨진 니즈 발견                                             │
│  • 데이터 기반 인사이트                                          │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 3: CONCEPT & STRATEGY                       8-15p (12%) │
│  ─────────────────────────────────────────────────────────────  │
│  • 핵심 컨셉 제시                                               │
│  • 차별화 전략                                                  │
│  • 경쟁 우위 포인트                                             │
│  • 접근 방법론                                                  │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 4: ACTION PLAN ★핵심                       30-60p (40%) │
│  ─────────────────────────────────────────────────────────────  │
│  • 상세 실행 계획                                               │
│  • 채널별 전략 (마케팅/PR의 경우)                                 │
│  • 실제 콘텐츠 예시 (비주얼 설명, 카피)                           │
│  • 캠페인 기획 및 일정                                           │
│  • KPI 및 측정 방법                                             │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 5: MANAGEMENT                               6-12p (10%) │
│  ─────────────────────────────────────────────────────────────  │
│  • 프로젝트 조직 구성                                           │
│  • 운영 프로세스                                                │
│  • 품질 관리 체계                                               │
│  • 리포팅 및 커뮤니케이션                                        │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 6: WHY US                                   8-15p (12%) │
│  ─────────────────────────────────────────────────────────────  │
│  • 수행 역량                                                    │
│  • 유사 프로젝트 실적                                           │
│  • 레퍼런스 및 사례                                             │
│  • 팀 소개                                                      │
├─────────────────────────────────────────────────────────────────┤
│  PHASE 7: INVESTMENT & ROI                          4-8p (6%)  │
│  ─────────────────────────────────────────────────────────────  │
│  • 투자 비용 구조                                               │
│  • 정량적 효과 예측                                             │
│  • ROI 분석                                                     │
│  • 부가 제안                                                    │
└─────────────────────────────────────────────────────────────────┘
                        총 70-140p (프로젝트 규모에 따라 조정)
```

### 6.2 제안서 유형별 Phase 가중치

제안서 유형에 따라 각 Phase의 비중이 자동으로 조정됩니다.

| Phase | Marketing/PR | Event | IT/System | Public | Consulting |
|-------|:----------:|:-----:|:---------:|:------:|:----------:|
| 0. HOOK | **8%** | 6% | 3% | 3% | 5% |
| 1. SUMMARY | 5% | 5% | 8% | 8% | 8% |
| 2. INSIGHT | 12% | 8% | 12% | 15% | 15% |
| 3. CONCEPT | 12% | 10% | 10% | 10% | 12% |
| **4. ACTION** | **40%** | **45%** | 35% | 30% | 30% |
| 5. MANAGEMENT | 8% | 10% | 12% | 12% | 10% |
| 6. WHY US | 10% | 10% | 12% | **15%** | 12% |
| 7. INVESTMENT | 5% | 6% | 8% | 7% | 8% |

**핵심 포인트**:
- **마케팅/PR**: ACTION PLAN(40%) + 실제 콘텐츠 예시가 핵심
- **이벤트**: ACTION PLAN(45%)이 가장 높음 (상세 실행 계획 중요)
- **공공/입찰**: WHY US(15%)가 높음 (실적 증명 중요)
- **IT/시스템**: SUMMARY(8%) + MANAGEMENT(12%)가 높음 (기술적 명확성)
